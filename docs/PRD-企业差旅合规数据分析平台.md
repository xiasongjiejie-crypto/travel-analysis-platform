## 2. 项目背景

企业差旅费用是可控成本中占比高、发生频次高、审核难度大的一类支出。传统的人工抽查审核方式存在以下痛点：

- **覆盖率低**：人工只能抽查少量单据，绝大多数违规无法被发现。
- **模式单一**：只能发现"超标"这类显性违规，对拆分报销、司乘合谋、时空矛盾等隐蔽违规无能为力。
- **滞后性强**：事后审计，无法在报销发生时实时拦截。
- **缺乏画像**：无法沉淀员工/商户/司机的历史行为，难以识别系统性风险。

本项目旨在构建一个**数据驱动的差旅合规分析平台**，通过多源数据交叉验证 + 规则引擎 + 异常检测 + 关联分析，实现差旅费用的自动化合规审查、风险评分与预警。核心思想是：**用"对公行程/预订"作为合规基线，去撞击"实际轨迹/消费/退改"数据，让违规无处遁形。**

## 3. 项目目标

### 3.1 业务目标
- 提升违规发现率，覆盖机票、酒店、用车、餐饮、报销全链路。
- 从"事后抽查"升级为"全量筛查 + 实时预警"。
- 沉淀员工、商户、司机的行为画像，识别系统性/团伙性违规。

### 3.2 产品目标
- 提供可配置的**规则引擎**，业务人员可自定义合规规则与阈值。
- 提供**风险评分与分级**，帮助审核人员聚焦高风险单据。
- 提供**告警与复核工作流**，形成"发现—复核—处置"闭环。
- 提供**合规看板与报表**，支撑管理决策。

### 3.3 衡量指标（成功标准）
- 违规检出率相较人工抽查提升 ≥ 3 倍。
- 硬规则告警准确率 ≥ 95%，软规则复核命中率 ≥ 30%。
- 单批次全量数据分析耗时可接受（满足 T+1 或近实时）。

## 4. 名词术语

| 术语 | 定义 |
|------|------|
| 差标 | 差旅标准，按员工职级、城市分级设定的费用上限 |

| 时空矛盾 | 同一员工在时间重叠的两笔消费地理距离超过可移动范围 |

## 6. 五大分析维度（核心分析框架）★

平台以「对公数据（差旅申请、企业预订）」为基线，撞击「实际数据（轨迹、消费、退改）」，从五个维度识别违规。每个维度下已落地为可执行规则（规则 ID 见附录 A）。

### 6.1 维度一：时空偏离度（空间与时间交叉验证）
比对员工的"对公行程"与"实际轨迹"，捕捉时空异常。
- **时空断层（幽灵行程）**：报销 A 地酒店，同一天打车/餐饮定位却在 B 地。 → `R-ST-01`
- **提前/延后停留（公私混同）**：机票/高铁买在批准差旅时间之外，周末多出市内消费或住宿。 → `R-ST-02`
- **时空分身（同城异地/闪现）**：短时间（如 30 分钟内）在距离过远的两地同时消费，往往是发票借给亲友使用。 → `R-ST-03`

### 6.2 维度二：消费反常度（同业与历史基线对比）
建立"常模"（正常消费区间），超出即标红。
- **精准压线**：连续多次报销金额精准等于标准上限（如均为 500 元），疑与商家勾兑开高票/套现。 → `R-CONS-01`
- **同行人消费割裂**：同期同地同行两人消费差 3 倍以上。 → `R-CONS-02`
- **本地高频消费**：差旅在外却在常驻地高频超市/加油大额消费，疑用日常发票冲抵。 → `R-CONS-03`

### 6.3 维度三：订退猫腻度（预订与退改操作审计）
违规常发生在"预订"与"退改"的操作链路里。
- **阴阳票据（买贵退便宜）**：预订全价后改签低折扣舱位，却按原价报销。 → `R-BK-01`
- **高频退改签**：退改频率显著高于常态，疑制造报销凭证或冲抵手续费。 → `R-BK-02`
- **双重套现（两头报销）**：企业账户已支付，事后又以发票/行程单个人报销。 → `R-BK-03`

### 6.4 维度四：关系关联度（人际网络与利益输送）
基于社交网络/知识图谱分析"谁和谁在一起消费"。
- **拆单（化整为零）**：同商户同时段多张发票各自低于审批上限但合计超限，规避高层审批。 → `R-REL-01`
- **出差搭子异常**：不同部门、业务无交集的员工订单在时间地点高度重合，疑借公出游。 → `R-REL-02`
- **发票连号**：报销的多张发票号连续，疑一次性索取整沓空白发票自填。 → `R-REL-03`

### 6.5 维度五：场景矛盾度（常识与逻辑校验）
用逻辑规则撞击违反常理的报销组合。
- **油费与打车矛盾**：报销自驾油费，同期又高频打车，疑虚构自驾套油补/私车油票冲账。 → `R-SC-01`
- **机票酒店断档**：买了去某地的机票、停留数日，却无住宿也无返程记录，疑现金住宿虚报补助。 → `R-SC-02`

## 8. 功能需求详述

### 8.1 数据接入与清洗（F-DATA）
- **F-DATA-01 多源数据接入**：差旅申请单、企业预订订单（携程等）、实际消费（滴滴/美团/发票）、用车、员工差标（CSV / 数据库 / API）。
- **F-DATA-02 数据标准化**：统一字段、时间、金额、地理坐标、城市名。
- **F-DATA-03 数据关联**：以 `employee_id`、`travel_apply_id`、`invoice_no`、`order_id` 等为键关联多源数据。
- **F-DATA-04 数据质量校验**：缺失值、异常值、重复记录检测。

### 8.2 规则引擎（F-RULE）
- **F-RULE-01 规则配置**：规则启用/停用、阈值、适用范围可配置。
- **F-RULE-02 五维度 + 三层规则体系**：规则归属五大维度，并标注层级（硬/软/模型）。
- **F-RULE-03 白名单机制**：通勤班车、驻场用车等场景白名单。
- **F-RULE-04 规则命中记录**：每条告警记录规则、维度、依据，可追溯。

> 具体规则清单见 **附录 A：合规规则库**。

### 8.3 司乘亲密度分析（F-AFFINITY）
- **F-AFFINITY-01 集中度计算**：最高频车牌行程占比（重复率 R）。
- **F-AFFINITY-02 HHI 指数**：识别绑定 2~3 个固定司机的情况。
- **F-AFFINITY-03 概率显著性检验**：基于城市司机密度计算期望重复次数，自动适配城市规模。
- **F-AFFINITY-04 最小样本门槛**：行程数低于阈值不判定，降低小样本误报。

### 8.4 风险评分与分级（F-SCORE）
- **F-SCORE-01 单据风险分**、**F-SCORE-02 员工风险画像**、**F-SCORE-03 风险分级**、**F-SCORE-04 多信号叠加升级**。

### 8.5 告警与复核工作流（F-FLOW）
- 告警生成、复核处置、处置反馈优化、通知。

### 8.6 合规看板与报表（F-DASH）
- 合规总览、维度下钻、高风险榜单、报表导出。

## 11. 司乘亲密度算法说明

### 11.1 集中度指标
```
重复率 R = 最高频车牌的行程数 / 该乘客总行程数
命中条件：总行程数 ≥ N 且 R ≥ 阈值
```
### 11.2 HHI 赫芬达尔指数
```
HHI = Σ (每个车牌行程占比)²   （越接近 1 越集中）
```
### 11.3 概率显著性检验（降误报）
```
期望重复次数 E ≈ n·(n-1) / (2M)     n=行程数, M=城市活跃司机数
实际重复 ≥ 2 且 远大于 E（>阈值倍数）判定显著异常
```
### 11.4 防误报要点
最小样本门槛、按城市司机密度动态调整期望、通勤/驻场白名单。

## 附录 A：合规规则库（五大维度 · 已落地 14 条）

> 字段：规则ID / 规则名称 / 检测逻辑 / 所需数据 / 层级

### A.1 时空偏离度

| 规则ID | 规则名称 | 检测逻辑 | 所需数据 | 层级 |
|--------|---------|---------|---------|------|
| R-ST-01 | 幽灵行程(时空断层) | 同一员工同一天消费出现 ≥2 个不同城市 | ConsumptionRecord | 模型 |
| R-ST-02 | 提前/延后停留(公私混同) | 机票/火车返程日期晚于批准差旅结束日 | BookingOrder × TravelApply | 软 |
| R-ST-03 | 时空分身(异地闪现) | 同员工两笔消费时间间隔<30分钟但城市不同 | ConsumptionRecord | 模型 |

### A.2 消费反常度

| 规则ID | 规则名称 | 检测逻辑 | 所需数据 | 层级 |
|--------|---------|---------|---------|------|
| R-CONS-01 | 精准压线 | 员工报销金额精准等于差标上限且出现 ≥2 次 | Reimbursement × TravelPolicy | 软 |
